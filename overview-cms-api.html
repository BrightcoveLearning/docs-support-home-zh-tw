一般信息

基本網址

的基本 URL 內容管理系統 API是：

        https://cms.api.brightcove.com/v1

帳戶路徑

在所有情況下，都將針對特定的視頻雲帳戶。因此，您將始終添加術語accounts後跟您的帳戶 ID 到基本 URL：

        https://cms.api.brightcove.com/v1/accounts/{account_id}

驗證

請求的身份驗證需要授權頭：

        Authorization: Bearer {access_token}

這access_token是一個臨時的 OAuth2 訪問令牌，必須從Brightcove OAuth服務。有關更多詳細信息，請參見Brightcove OAuth概述。

創建客戶端憑據的最簡單方法是通過 Brightcove Studio 管理頁面。有關詳細說明，請參見管理API身份驗證憑據

運作方式

當您請求客戶端憑據時，您需要指定所需的帳戶訪問或操作類型。以下是當前支持的操作列表內容管理系統 API：

  • 視頻數據：

    video-cloud/video/read
    video-cloud/video/create
    video-cloud/video/update
    video-cloud/video/delete
    或者
    video-cloud/video/all
    video-cloud/video/assets/delete (Only needed if you want to delete digital masters - note that you cannot get this permission when create credentials in Studio.必須通過OAuth API或Brightcove學習服務創建的示例應用程序）

  • 播放清單資料：

    video-cloud/playlist/read
    video-cloud/playlist/create
    video-cloud/playlist/update
    video-cloud/playlist/delete
    或者
    video-cloud/playlist/all

  • 通知：
    • video-cloud/notifications/all (for Notifications)

限速

這個內容管理系統 API提供對數據的未緩存讀取訪問。這對於時間敏感的發布工作流程很重要，因為您使用內容管理系統 API并快速讀取數據以驗證其正確性。

這內容管理系統 API不適合大規模運行時使用（例如訪問高流量公共網頁上的視頻）。對於高流量應用程序，您必須使用緩存接口，例如：播放接口、圖庫、播放器或本機 SDK。

為確保性能視頻雲系統，不超過 20 個並發調用內容管理系統 API每個帳戶允許。訪問頻率應小於每秒10個請求。

當並行運行的多個進程或應用程序發出 API 請求時，會發生並發調用。由於完成一個請求可能需要一秒鐘以上的時間，因此即使每秒發出的請求不超過 10 個，也可能會超出並發限制。

如果將多個應用程序集成到內容管理系統 API對於一個帳戶，那麼這些應用程序應該具有回退和重試邏輯來處理達到並發限製或速率限制的實例。

如果超出了速率或併發限制，則429 - TOO_MANY_REQUESTS錯誤將被返回。

參考編號衝突

為了確保引用 ID 的唯一性，內容管理系統 API在對其分配的視頻進行任何操作後，將 ID 鎖定最多 3 分鐘。當您嘗試重試失敗過快的請求時，或者在刪除之前分配給它的視頻後過早嘗試重新使用參考 ID 時，這可能會導致返回 409 錯誤。見錯誤信息參考更多細節。

視頻資產限制

每個視頻的資產限制為 1,000 個。資產包括演繹，音頻軌道，文本軌道和圖像，以及數字母帶。字幕和圖像的總資產很少會超過10-15，因此即使您擁有針對150種不同語言的單獨音軌和文本音軌，資產也仍然少於350。

使用注意事項

這內容管理系統 API用於集成和發布工作流。您不能從公共網頁對此 API 進行客戶端調用，因為這樣做會涉及為 API 公開您的客戶端憑據（因此，該 API 未啟用 CORS）。您可以從客戶端 (Web) 應用程序進行調用，前提是您通過服務器端代理路由 API 調用。Brightcove 學習服務提供了幾個示例應用程序使用這種方法的，以及構建此類應用程序的指南 .

如果您要檢索所有視頻或大量視頻，請務必查看關於大數據集的說明。

避免使用硬編碼的URL

縮略圖、海報、視頻文件和其他媒體的 URL 不應在您的頁面或應用程序中進行硬編碼。這內容管理系統 API將始終返回媒體文件的當前 URL，但 URL 本身可能會更改。你應該使用內容管理系統 API（ 或者播放接口 ) 請求在每次頁面加載時檢索這些 URL，或將它們緩存不超過 6 小時。

緩存視頻和圖像URL

您可以緩存視頻和圖像的 URL 以提高頁面性能，但必須定期刷新緩存。如果您緩存檢索到的URL以提高頁面性能，請確保通過每六個小時至少重複一次API調用來刷新緩存。

方法

目前，API支持以下請求類型：

  • GET
  • POST
  • PATCH
  • PUT
  • DELETE

參數

請注意，所有參數都是可選的。除非另有說明，否則它們適用於GET要求提供視頻和播放列表。

GET請求參數
範圍 描述
q 查詢字符串以進行搜索-請參閱搜索語法想要查詢更多的信息
limit 要返回的視頻數量-必須為1到100之間的整數。默認：20
offset 要跳過的視頻數量（用於分頁結果）。必須為正整數。默認：0
sort 一個字符串，指定要作為排序依據的字段。從...開始-降序排序。如果為q提供，則默認排序是按“分數"（搜索結果與原始查詢的相關性）。如果沒有價值q提供，則默認排序為updated_at下降。以下字段對排序有效：name , reference_id , created_at , published_at , updated_at , schedule_starts_at , schedule_ends_at , state , plays_total， 和plays_trailing_week

搜索視頻

布賴特科夫的內容管理系統 API提供了一種在您的視頻中搜索視頻的程序化方式視頻雲圖書館。

要對您的視頻數據執行基本和復雜的搜索，您將使用q範圍：

        https://cms.api.brightcove.com/v1/accounts/921483702001/videos?q={search terms}

有關如何搜索視頻的詳細信息，請參見搜尋影片文檔。

對於播放列表，搜索字符串的支持值受到更多限制。您目前可以通過以下方式搜索type , name , description， 和reference_id .以下是一些有效搜索示例：

  • q=type:EXPLICIT
  • q=type:ACTIVATED_OLDEST_TO_NEWEST
  • q=type:ALPHABETICAL
  • q=bears+otters (name, description, or reference ID must contain either "bears" or "otters")
  • q=%2Bname:bears+type:EXPLICIT (name must contain "bears")

看搜索播放列表更多細節。

注意搜索詞playable:true/false您可以使用它來僅返回可播放（或不可播放）的視頻： 
        https://cms.api.brightcove.com/v1/accounts/:account_id/videos?q=%2Bplayable:true

此查詢要求返回的視頻是可播放的（未停用、未預定且不缺少可播放的再現）。

如果您要搜索的是不是可玩，使用q=%2Bplayable:false。

分頁結果

使用 limit 參數來指定您要在請求中返回多少個項目 - 最多 100 個。然後您可以使用 offset 用於翻閱大於的結果集的參數limit .這offset是要跳過的項目數。

例如，以下搜索返回總結果集中的視頻51-75，假設總結果集中至少有75個視頻：

        /videos?q=updated_at:2014-01-01..2014-06-30&limit=25&offset=50

這limit和offset參數可用於視頻和播放列表。

筆記：limit和offset做不是當您請求屬於播放列表的視頻時工作 - 此請求將始終返回播放列表中的所有視頻。

分頁最佳做法

由於CMS API可能正在進行並發修改操作，因此在分頁結果集時建議遵循以下步驟：

  1. 做一個 count 請求獲取您的結果集中的視頻總數。 
          /accounts/578380111111/counts/videos?q=tags:nature
  2. 使用 limit 和 offset 從結果集中返回數據組的參數。 
          /accounts/578380111111/videos?q=tags:nature&limit=20&offset=50
  3. 請注意，某些頁面上的視頻可能少於20個。當您要求與第一個一樣多的視頻時，您將知道您已到達結果集的末尾count要求。

總而言之，繼續檢索頁面，直到獲得與原始視頻數量相等的視頻數量count請求，因為這個數字應該是高估的錯誤。要求：

      count / page-size + 1 page

排序視頻結果

使用參數sort=field_name指定應如何對結果進行排序 - 您可以對視頻和播放列表進行排序。您可以對以下視頻字段進行排序： [1-1]

使用減號 ( sort= -field_name ) 為降序。
  • name
  • reference_id
  • created_at
  • published_at
  • updated_at
  • schedule_starts_at (note: this is the sort field - the search field is schedule.starts_at )
  • schedule_ends_at (note: this is the sort field - the search field is schedule.ends_at )
  • state
  • plays_total [1-2]
  • plays_trailing_week [1-2]

筆記

  • [ 1-1] 如果您沒有為視頻搜索呼叫提供排序值，結果將按相關性排序。如果您不提供GET視頻通話，結果將按以下順序排序updated_at下降。
  • [ 1-2] 你可以排序plays_total或者plays_trailing_week，但這些字段不包含在結果中

排序播放列表結果

您可以按以下字段對播放列表進行排序：

  • name
  • updated_at (default)
使用減號 ( sort= -field_name ) 為降序。

所有視頻和大數據集

如果您要檢索帳戶中的所有視頻或大量視頻，則必須注意以下幾點：

  1. 您可能想使用允許的最大limit (100)，但最好以 25 個或更少的批次檢索視頻，以盡量減少 API 請求超時的可能性
  2. 在翻閱大型數據集時，可能會在操作過程中更新視頻數據，這可能導致項目的響應發生偏移：
    • 您可能會在連續頁面上看到重複的項目
    • 某項可能已丟失，因為它已轉移到先前的響應集

    為了解決第一種可能性，您的應用程序應該在您完成檢索視頻後刪除完整的項目列表。要處理第二種可能性，您需要將檢索到的項目總數（去重後）與您期望的數量進行比較，然後重新運行請求，按 last_modified_date（降序）對結果進行排序 - 您不應該需要檢索多於一批以撿拾遺漏的物品。

  3. 您可以通過對返回的結果進行適當排序來降低上一項中出現場景的可能性。默認排序方式關聯 for search 基於一種複雜的算法，該算法看起來是關鍵字、標籤和自定義字段值的組合。如果您要搜索基於多個關鍵字，標籤和/或自定義字段的視頻，則按相關性排序正是您想要的。但是，如果您只是嘗試檢索所有或大量視頻，請設置sort參數顯式將使您可以更好地控制返回項目的順序。

  • 創建，檢索，更新和刪除視頻

  • 獲取視頻源，圖像和數字主信息
  • 獲取視頻所屬的播放列表

有關視頻操作的詳細信息，請參見API參考。

資產

資產操作使您可以管理資產，包括格式，清單，圖像和文本軌道。要攝取資產，您必須使用動態攝取 API .這POST和PATCH操作為內容管理系統 API /assets可用於添加和更新遠程資產 .內容管理系統 API GET 操作適用於兩個都攝取和遠程資產。

  • 添加，更新或刪除演繹
  • 添加，更新或刪除數字母版的元數據
  • 添加，更新或刪除分段視頻類型（例如HLS）的清單
  • 添加，更新或刪除海報和縮略圖
  • 添加，更新或刪除WebVTT文本軌道或DFXP字幕

有關資產操作的詳細信息，請參見API參考。

通知

您可以在以下時間收到通知video_change事件發生在您的視頻庫中。通知將發送到您指定的URL，該URL應指向能夠處理的應用程序HTTP POST要求。

筆記：
  • 視頻元數據的任何更改都會觸發video_change事件。視頻使用的資產的更改將不是觸髮變更事件。
  • 除了通知外，還有狀態API可讓您獲取與視頻相關的提取（嵌套，替換或重新轉碼）作業的狀態。

通知失敗

通知系統將從客戶服務器返回的任何 4xx 或 5xx 視為可重試故障。失敗的回調將最多重試20次，隨後的回調之間的延遲將成倍增加。前幾次重試將在初始回調嘗試後的幾分鐘內發生。如果回調繼續失敗，並且我們一直進行到第20次重試，則重試延遲將是幾天。

video_change events are triggered by automated system processes as well as user actions, so these events will not always corresponds to changes you made to video properties in Studio or via the APIs.

防火牆

如果您的組織對通過防火牆的傳入流量來源有嚴格的政策，我們會允許所有AWS East地區IP。這可能會發生變化，因此所有 AWS IP 都應列入白名單。看https://docs.aws.amazon.com/general/latest/gr/aws-ip-ranges.html想要查詢更多的信息。
